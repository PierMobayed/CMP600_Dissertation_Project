## 1. Ethics (обхват и ограничения)

- **Данни:** само симулирани поръчки/пратки; няма реални имена, адреси или PII.
- **Потребители:** няма полеви участници извън одобрения етичен обхват; вътрешен технически тест (ако го споменаваш) не събира поведенчески данни извън логовете на прототипа.
- **Съхранение:** пароли в прототипа са плоски (демонстрационни) — посочи като **риск/limitation** и че production би изисквал хеширане.
- **Клауза за AI (ако е приложима):** декларирай използване на инструменти за писане/код според правилата на институцията.

## 2. Nielsen heuristics (наблюдения от dashboard / клиент / шофьор)

За всеки принцип: **Observation → Severity (0–4) → Recommendation.**

| ID | Принцип | Бележки за MVP |
|----|---------|----------------|
| H1 | Видимост на статуса | Примери: ETA, статус на пратка, KPI на dashboard. |
| H2 | Съответствие с реалния свят | „Shipment“, „driver“, „delayed“ — домейн език. |
| H3 | Контрол и свобода | Изход, филтри, смяна на job при шофьора. |
| H4 | Консистентност | Един стил на бутони/етикети между трите приложения (или отклонения). |
| H5 | Помощ при грешки | Съобщения при login/register; 401 от API. |
| H6 | Разпознаване вместо припомняне | Карта + timeline на събития вместо само ID-та. |
| H7 | Гъвкавост | Прототип акаунти; филтри по статус/шофьор/дата. |
| H8 | Естетика и минимализъм | KPI не претрупват (Few-style dashboard). |
| H9 | Помощ при грешки recovery | … |
| H10 | Помощ и документация | `/docs`, README, batch старт. |

## 3. SUS (ако използваш System Usability Scale)

- **Въпросник:** стандартните 10 твърдения (Likert 1–5) × поне една роля (напр. „dispatcher“ за dashboard).
- **Изчисление:** сума с теглови коефициенти → SUS 0–100 (потърси официална формула).
- **Интерпретация:** ориентировъчно &gt;68 = над средното, но цитирай методологията си.

*Ако SUS не е одобрен в етиката, остани при heuristics.*
