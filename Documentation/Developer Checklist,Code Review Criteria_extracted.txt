✅ CODE REVIEW CRITERIA (ACCEPT / REJECT RULES)
If a change fails ANY critical criterion → it is rejected.
1. Functional Correctness (CRITICAL)
✅ ACCEPT if:
Implements requirements exactly
Matches API contract
Produces correct output with simulated data
❌ REJECT if:
Feature partially works
Endpoint differs from API contract
Dummy or placeholder logic remains
2. Stability & Reliability (CRITICAL)
✅ ACCEPT if:
No crashes on basic flows
Errors handled gracefully
App recovers from minor issues
❌ REJECT if:
App crashes
Unhandled exceptions
Backend fails under light load
3. Simplicity & Explainability (CRITICAL for Viva)
✅ ACCEPT if:
Code can be explained in plain English
Logic is straightforward
No unnecessary abstractions
❌ REJECT if:
Overengineering
Code “just works” but can’t be explained
Complex patterns with no benefit
4. Ethics & Data Compliance (CRITICAL)
✅ ACCEPT if:
Only simulated data used
No personal data stored
No real user accounts
❌ REJECT if:
Any personal or identifiable data
External real APIs used without approval
5. UI & Usability (HIGH PRIORITY)
✅ ACCEPT if:
UI is clear
Dashboard not cluttered
Text is readable
❌ REJECT if:
Confusing layout
Too much information
Poor visibility
6. Performance (HIGH PRIORITY)
✅ ACCEPT if:
API latency reasonable (<300ms typical)
Dashboard updates smoothly
GPS updates do not overload backend
❌ REJECT if:
Noticeable lag
Dashboard freezes
Blocking operations
7. Maintainability (MEDIUM PRIORITY)
✅ ACCEPT if:
Code structure is logical
Files are named clearly
Functions are small and focused
❌ REJECT if:
Spaghetti code
Giant functions
Unclear naming
8. Documentation Support (MANDATORY)
✅ ACCEPT if:
Developer provides short explanation of feature
Screenshots included when required
❌ REJECT if:
No explanation
No evidence for dissertation/Viva
✅ FINAL RULE (VERY IMPORTANT)
If a feature looks impressive but risks Viva stability → REJECT it.
This is an academic project.Clarity > ComplexityStability > Features
